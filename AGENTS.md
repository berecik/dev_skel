# Agents Rules

Authoritative rules and onboarding guide for all AI agents (LLM assistants)
working on the `dev_skel` project. This file complements `_docs/JUNIE-RULES.md`
and `_docs/LLM-MAINTENANCE.md`. When rules differ, follow the most restrictive
guidance unless the user explicitly overrides it.

`/CLAUDE.md` mirrors this file for Claude Code and adds Claude-specific
operational notes; the rules below apply to all agents regardless of which
entrypoint loads them.

---

## 1. Read These Files First (in order)

1. `/AGENTS.md` (this file) — or `/CLAUDE.md` if you are Claude Code
2. `_docs/JUNIE-RULES.md` (project-authoritative rules)
3. `_docs/LLM-MAINTENANCE.md` (operational guide and workflows)

load:

5. `_skels/<name>/AGENTS.md` (if present) or `_skels/<name>/CLAUDE.md`
6. `_skels/<name>/JUNIE-RULES.md`
7. The skeleton page in `_docs/` (e.g. `_docs/python-fastapi-skel.md`)

Refresh these files whenever the user says rules/docs changed or after you make
rule-affecting edits.

## 2. Project Purpose and Your Role

- **Dev Skel generates whole multi-service projects where every service
  ships with its own in-service code agent.** Three AI surfaces (each
  with its own tools, smokes, and design doc):
  1. `_bin/skel-gen-ai` — full-stack dialog → 5-phase Ollama
     orchestration (per-target overlay × 2 → integration → test-and-fix
     → docs). Backed by `_bin/skel_rag/` (FAISS RAG agent) under the
     `_bin/skel_ai_lib.py` shim.
  2. `./ai` — per-service code agent shipped in every generated
     service (`propose` / `apply` / `verify` / `explain` / `history` /
     `undo` / `upgrade`). In-tree mode = full RAG; out-of-tree mode =
     stdlib-only. Wrapper-level `./ai` fans out across services by
     default.
  3. `./backport` (service → template, bumps `<skel>/VERSION` and
     appends `<skel>/CHANGELOG.md`) and `./ai upgrade` (template →
     service, replays the changelog excerpt as a propose/apply
     request).
- Makefile-based project generator system managing 10 skeletons in
  `_skels/*-skel/`. Each skeleton ships a static `gen` scaffolder, a
  `merge` rsync helper, an AI manifest at
  `_skels/_common/manifests/<skel>.py`, a `VERSION` + `CHANGELOG.md`,
  and per-stack `test` / `build` / `run` / `install-deps` scripts.
- Each skeleton produces a framework-specific service inside a wrapper project.
  The wrapper integrates services around a shared environment file
  (`<wrapper>/.env`) plus a shared SQLite/Postgres database
  (`<wrapper>/_shared/db.sqlite3` by default) so multiple services can be
  generated into the same wrapper without conflict.
- **Service directories are named after the service display name.** Slug
  computation lives in `dev_skel_lib.slugify_service_name()`. The CLI
  positional order is `_bin/skel-gen [proj_name] [skel_name] [service_name]`
  (and the same for `skel-gen-ai` / `skel-gen-static`); all three are
  optional. The third positional arg is the **display name** (e.g.
  `"Ticket Service"`); the on-disk dir is its slug. When omitted, the
  per-skel default (`backend` / `frontend` / `service`) is used for
  backwards compat. Pass `.` (or omit) as `proj_name` to install into
  the current directory; missing `skel_name` falls into an interactive
  picker (in static mode) or the full-stack dialog (in AI mode).
- **`_bin/skel-gen` defaults to AI (as of 2026-04).** It exec's
  `_bin/skel-gen-ai` unless `--static` is passed (or the user invokes
  `_bin/skel-gen-static` directly). Scripted callers that want a
  deterministic AI-free baseline must use `--static` or
  `_bin/skel-gen-static`.
- **`_bin/skel-add`** is the existing-project sibling of `skel-gen`.
  It uses the same positional order (`[proj_name] [skel_name]
  [service_name]`) but requires the wrapper to exist already. Default
  mode is AI (`_bin/skel-gen-ai --skip-base`) so the new service still
  goes through the integration/test-fix/docs phases with the already
  generated sibling services; `--static` dispatches to
  `_bin/skel-gen-static --existing-project`.
- The wrapper ships multi-service dispatch scripts (`./services`, `./run`,
  `./test`, `./build`, `./stop`, `./install-deps`) generated by
  `_skels/_common/common-wrapper.sh`. Every script sources the wrapper
  `.env` and accepts an optional first arg matching a service slug.
- The root `Makefile` orchestrates `gen-*` and `test-*` targets; `_bin/skel-gen`
  is a relocatable Python entrypoint that delegates to skeleton-level `gen`
  scripts.
- All `_bin/` CLI tools (`skel-install`, `skel-update`,
  `skel-sync`, `skel-list`, `skel-gen`, `skel-gen-ai`, `skel-add`) are Python and
  share helpers through `_bin/dev_skel_lib.py`. Treat that module as the
  single source of truth for config loading, project generation, slug
  computation, and rsync wrappers.
- `_bin/skel-gen-ai` is the AI-augmented sibling of `skel-gen`. It runs the
  interactive dialog FIRST (service name, item, auth), uses the chosen
  service name's slug as the directory, and then asks Ollama (via
  the `_bin/skel_rag/` RAG agent, behind the legacy `_bin/skel_ai_lib.py`
  shim) to rewrite the files listed in `_skels/_common/manifests/<skel>.py`.
  **All 10 skeletons** are now AI-supported — Python (django, django-bolt,
  fastapi, flask), Java (spring), Rust (actix, axum), Next.js,
  React (ts-react), and Flutter (flutter). The picker
  (`_bin/skel-gen-ai` with no arguments) auto-discovers manifests, and
  the full-stack dialog auto-partitions backends vs frontends via
  `_SERVICE_KIND_BY_TECH` so React and Flutter both appear in the
  frontend list. **Defaults** (since 2026-04): the backend picker
  defaults to the first skel that ships the wrapper-shared items API;
  the **frontend picker defaults to `ts-react-skel`**. **Both halves
  are optional** — pass `--no-frontend` for a backend-only project,
  `--no-backend` for a frontend-only project. The dialog refuses the
  "neither chosen" combination.
- **AI generation goes through `_bin/skel_rag/`** (since 2026-04). The
  package indexes each skeleton's reference templates with tree-sitter
  + FAISS so prompts retrieve only the most relevant chunks instead of
  stuffing whole files. `_bin/skel_ai_lib.py` is now a thin
  backwards-compat shim that re-exports every public symbol and
  delegates orchestration (`generate_targets`, `run_integration_phase`,
  `run_test_and_fix_loop`, `_ask_ollama_to_fix`) to
  `skel_rag.agent.RagAgent`. The chat call goes through
  `langchain_ollama.ChatOllama` instead of the previous urllib client.
  Manifests can opt into the new `{retrieved_context}` /
  `{retrieved_siblings}` placeholders (FastAPI is the migrated
  reference); manifests still using `{template}` / `{wrapper_snapshot}`
  keep working unchanged. Install the heavy deps once with
  `make install-rag-deps`. The standalone debug CLI is `_bin/skel-rag`
  (subcommands: `index`, `search`, `info`, `clean`). Full docs in
  `_docs/LLM-MAINTENANCE.md` under "`_bin/skel_ai_lib.py` (legacy
  shim) + `_bin/skel_rag/` (RAG agent)".
- **Every backend skel** — Python (django, django-bolt, fastapi, flask),
  Java (spring), Rust (actix, axum), and JS (node) — reads
  `DATABASE_URL` / `JWT_SECRET` (and the JVM-friendly
  `DATABASE_JDBC_URL` / `SPRING_DATASOURCE_URL` siblings) from the
  wrapper-level `.env` first, then its own local `.env`. Per-stack
  contract:
  - Python: `python-dotenv` + helper in `settings.py` / `core/config.py`.
  - Java: `${...}` placeholders in `application.properties` plus the
    `JwtProperties` `@ConfigurationProperties` bean.
  - Rust: `src/config.rs::Config::from_env()` + `load_dotenv()`.
  - JS: `src/config.js` with the `dotenv` package.
  Never hardcode a sqlite path or JWT secret in skel templates — the
  env-driven flow is the contract that makes services interchangeable
  inside a wrapper. Keep the three database URL variants in lockstep
  when switching to Postgres.
- **Frontend skels** read the same wrapper-shared `.env`:
  - React (`ts-react-skel`): `vite.config.ts` loads `<wrapper>/.env`
    at build time and exposes a safe subset (`config.backendUrl`,
    `config.jwt.{algorithm,issuer,accessTtl,refreshTtl}`,
    `config.services`) via `src/config.ts`. `JWT_SECRET` is NOT
    promoted into the bundle.
  - Flutter (`flutter-skel`): `lib/config.dart` loads `<wrapper>/.env`
    at runtime via `flutter_dotenv` (the gen script copies the
    wrapper `.env` into the project as a bundled asset). The JWT
    bearer token is stored via `flutter_secure_storage`
    (Keychain / EncryptedSharedPreferences / native equivalents),
    and `AppConfig` deliberately omits `JWT_SECRET` so it never
    lands in a mobile bundle.
  Both frontends speak the wrapper-shared `/api/items` +
  `/api/auth/login` + `/api/state` contract that
  `python-django-bolt-skel` and `python-fastapi-skel` ship out of
  the box.
- After project generation, `skel-gen` renders `_skels/_common/AGENTS.md` (and
  any per-skeleton `AGENTS.md`) with placeholders such as `${project_name}`,
  `${service_dir}`, `${skeleton_name}`, and `${skeleton_doc}` so generated
  projects ship with up-to-date agent rules.

Your responsibilities:
1. Maintain generators and skeletons.
2. Keep dependencies healthy and reasonably current.
3. Preserve developer workflows and ergonomics described in `_docs/`.

`_test_projects/` contains generated outputs for e2e tests. Never hand-edit it.

## 3. Directory Responsibilities (quick map)

- `Makefile` (root): entry for generator/test targets.
- `_bin/`: Python CLI tools, the shared `dev_skel_lib.py` module, the
  Ollama-backed `skel_ai_lib.py` AI library, and the `rsync-*-excludes.txt`
  files used by install/update/sync.
- `_skels/_common/`: shared resources used by every skeleton — the
  `common-wrapper.sh` script that scaffolds wrapper-level files, the
  templated `AGENTS.md` / `CLAUDE.md` rendered into generated projects, and
  per-skeleton AI manifests under `_skels/_common/manifests/`.
- `_skels/*-skel/`: one directory per skeleton, each with `Makefile`, `gen`,
  `merge`, `test`, `deps`, and `install-deps`; should also provide
  skeleton-specific rules/docs (`AGENTS.md`, `JUNIE-RULES.md`).
- `_docs/`: documentation hub including `LLM-MAINTENANCE.md`, per-skeleton
  guides, and the dependency policy.
- `_test_projects/`: generated test apps only. Test Project Location Rule: any
  temporary/repro/debug projects you generate must live here, not elsewhere.

## 4. Rules Model and Updates

### 4.1 Global vs Per-Skeleton

- Global: `/AGENTS.md`, `_docs/JUNIE-RULES.md`, `_docs/LLM-MAINTENANCE.md`.
- Per-skeleton: `_skels/<name>/AGENTS.md` and/or `_skels/<name>/JUNIE-RULES.md`
  plus the skeleton doc in `_docs/`.

Apply all relevant global rules first, then overlay skeleton rules that do not
conflict. If conflicts arise, global rules win unless the user overrides them.

### 4.2 When to Update Rules

Update the appropriate rules/doc files whenever architecture, workflows, or
maintenance procedures change. Keep global changes in `/AGENTS.md` and
`_docs/JUNIE-RULES.md`, skeleton-specific changes in the matching skeleton
files, and operational steps in `_docs/LLM-MAINTENANCE.md`.

## 5. Operating Expectations (summary)

- Prefer minimal, surgical changes that solve the immediate issue.

## 6. Maintenance Workflow (default)

Unless the user specifies otherwise:
1. Run generator tests: `make clean-test` then `make test-generators`.
2. Fix all reported failures with minimal, targeted changes; re-run the same
   tests until green.
3. Only after tests are green, adjust rules/docs (global and per-skeleton) to
   reflect behaviour changes.
4. If maintaining a single skeleton, you may limit tests and doc updates to
   that skeleton plus any impacted globals.

## 6.3 Backporting Service Edits Into the Skeleton (`./backport`)

The sibling of `./ai` for the **service → template** direction.
Every generated service ships `./backport` (installed by the same
`install-ai-script` hook). Use it when the user has hand-edited or
`./ai`-edited files inside their service and wants the changes to
land in the skeleton so future generations inherit them.

Workflow:

1. `./backport` (or `./backport propose`) is a dry-run that lists
   every service file that differs from its same-relative skeleton
   file. Each candidate is `service_path → skeleton_path`; a JSON
   summary is written to `<dev_skel>/.ai/backport/<ts>-<sha>/`.
2. Review the list with the user.
3. `./backport apply` writes the changes back into the skeleton
   tree. The maintainer commits the skeleton diff manually.

REQUIRES a reachable dev_skel checkout (auto-detected via
`$DEV_SKEL_ROOT` → walk-up → common locations). When detached the
script exits 1 with an actionable error.

Sidecar: `./backport` reads `.skel_context.json` in the service
directory to know which skeleton to write to. Created by
`install-ai-script` when `common-wrapper.sh` runs; each `gen`
exports `SKEL_NAME=<basename $SKEL_DIR>` so the just-generated
service gets a force-overwritten sidecar.

Operator commands: `make test-backport-script` (propose+apply
round-trip smoke; modifies a file, applies, asserts skel updated,
reverts via `git checkout`); `_bin/skel-backport
propose|apply <service>` (maintainer CLI that `./backport`
forwards to).

Implementation: `_skels/_common/refactor_runtime/backport`,
`_bin/dev_skel_backport.py`, `_bin/skel-backport`,
`_bin/skel-test-backport-script`.

**Versioning side effect:** every successful `apply` bumps the
target skeleton's `<skel>/VERSION` (semver patch by default) and
prepends a Keep-a-Changelog entry to `<skel>/CHANGELOG.md`
listing each backported file. This is what lets `./ai upgrade`
(see § 6.5) know which changes to replay against an
already-generated service. Best-effort — write failures emit a
warning and do not abort the apply.

## 6.5 Versioning + `./ai upgrade` (template → existing service)

Every skel ships a `VERSION` file (semver) and a `CHANGELOG.md`
(Keep-a-Changelog format). The contract:

* `install-ai-script` writes the current `<skel>/VERSION` into the
  service sidecar as `skeleton_version` so each generated service
  remembers the template revision it was built from.
* `_bin/skel-backport apply` bumps `<skel>/VERSION` (patch by
  default) and prepends a `## [VERSION] - DATE` entry to
  `<skel>/CHANGELOG.md` listing each backported file. This makes
  the changelog an **auditable, replayable log** of every accepted
  service-to-template change.
* `./ai upgrade` (per-service) reads the sidecar's
  `skeleton_version`, compares to the skel's current `VERSION`,
  extracts the matching CHANGELOG entries, and synthesises an AI
  request that asks the model to "upgrade this service to vY by
  applying these changes". Dispatched through the standard
  propose/apply flow so all the safety machinery (git stash,
  fix-loop, verify) still applies. Dry-run by default; pass
  `--apply` to commit. On a successful apply the sidecar's
  `skeleton_version` is rewritten to the new baseline.
* Wrapper-level `./ai upgrade` fans out by default (see § 6.4),
  upgrading every service in the project at once.

Operator commands: `make test-ai-upgrade` (no LLM — covers the
no-op + outdated paths), `make test-ai-fanout` (no LLM — covers
the wrapper-level fan-out default for `./ai` and `./ai upgrade`).
Implementation: `_cmd_upgrade` + `_changelog_excerpt` /
`_semver_tuple` in `_bin/dev_skel_refactor_runtime.py`,
`_bump_skeleton_version` + `_bump_patch` in
`_bin/dev_skel_backport.py`.

## 6.4 Project-wide `./ai` (wrapper-level + cross-call memory)

Two layers turn the per-service `./ai` into a project-wide tool:

**Wrapper-level dispatcher** at `<wrapper>/ai`. Three modes:

* `./ai "REQUEST"` — fans out across every service (project-wide
  default since 2026-04).
* `./ai <service> "REQUEST"` — forwards to a specific service.
* `./ai --all "REQUEST"` — explicit fan-out (now redundant with
  the bare form, kept for backwards compatibility).

`./backport` is on the same fan-out list, so `./backport` /
`./backport apply` at wrapper level operates across every service
by default (mental model: "at the project level, talk to the whole
project; at the service level, talk to just this service").

**Cross-call memory** — every `./ai apply` appends one JSONL entry
to BOTH `<service>/.ai/memory.jsonl` AND `<wrapper>/.ai/memory.jsonl`.
The next `./ai propose` prepends the last 5 entries to the LLM
prompt as `## PREVIOUS_AI_RUNS`. The model gains continuity across
calls AND across services within the same wrapper. Memory writes
are best-effort (never block apply); reads silently skip malformed
JSONL lines.

`./ai history` prints both layers (project memory + local runs).

Operator command: `make test-ai-memory` (no LLM). Implementation:
`_bin/dev_skel_refactor_runtime.py` (memory helpers + post-apply
writes), `_skels/_common/common-wrapper.sh` (dispatch template +
`--all` recogniser).

## 6.2 Refactoring Inside a Generated Service (`./ai`)

Every generated service ships a `./ai` script (plus a vendored
`.ai_runtime.py`) installed by
`_skels/_common/refactor_runtime/install-ai-script` from
`common-wrapper.sh`. It is the **service → service** AI surface
(`skel-gen-ai` is template → service; `skel-backport` is
service → template).

Workflow:

1. Suggest `./ai "<one-sentence description>"` as a dry-run first
   (proposals land in `.ai/HEAD/`; the `.ai/` dir is git-ignored).
2. Review the diff with the user.
3. `./ai apply "..."` applies + runs the service test command read
   from `.skel_context.json` (or `./test`) and invokes the fix-loop
   on failure.
4. Other subcommands: `./ai history`, `./ai explain`,
   `./ai verify`, `./ai undo`.

Two activation modes are auto-detected: **in-tree** (FAISS via
`skel_rag.agent.RagAgent`) and **out-of-tree** (stdlib + Ollama HTTP
only).

Safety: every write is `_under(service_dir)`-checked, dirty git tree
requires `--allow-dirty`, pre-apply state is captured to a git
stash, a per-service `.ai/.lock` prevents concurrent applies, and
the parser rejects path-traversal / duplicate paths / empty bodies.

Operator commands: `make test-ai-script` (dispatch smoke, no LLM),
`make sync-ai-runtime` (after editing
`_bin/dev_skel_refactor_runtime.py`), `_bin/skel-ai SERVICE_DIR
"REQUEST"` (drive `./ai` from outside a service).

Implementation: `_bin/dev_skel_refactor_runtime.py` (canonical),
`_skels/_common/refactor_runtime/` (vendored copy + installer).
Design doc: `SERVICE_REFACTOR_COMMAND.md`.

## 7. Safety and Constraints

1. Avoid broad refactors without explicit user approval.
2. Prefer stable, well-supported dependency updates; avoid changes that risk
   widespread breakage.
3. If you cannot confirm external data (e.g., versions), keep current pins and
   note the limitation in docs/commits.
4. Maintain clarity and consistency with each module/skeleton’s existing style.
5. When in doubt about destructive/backward-incompatible actions, ask the user.

- Run relevant `make test-*` targets; for skeleton edits, include skeleton-level

- Confirm documentation/rules are consistent with the implemented behaviour.
- Leave `_test_projects/` untouched except via regeneration.
- After pushing, check CI status with `make ci-status` or `make ci-watch`
