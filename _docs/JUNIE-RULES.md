# Junie Project Rules

Authoritative rules and onboarding guide for Junie (and other LLM assistants)
working on the `dev_skel` project. This file is loaded **first** and sets the
baseline; apply skeleton-specific rules after this, never contradicting the
most restrictive guidance unless the user explicitly says so.

---

## 1. Files Junie Must Read First (in order)

1. `_docs/JUNIE-RULES.md` (this file)
2. `_docs/LLM-MAINTENANCE.md` (operational guidance and workflows)
3. `_docs/README.md` (documentation index)
4. `/AGENTS.md` (cross-agent baseline) — Claude Code reads `/CLAUDE.md`
   instead, which delegates to `/AGENTS.md`

When working on a specific skeleton (for example `python-fastapi-skel`), also
load:

5. `_skels/<name>/JUNIE-RULES.md`
6. The skeleton page in `_docs/` (e.g. `_docs/python-fastapi-skel.md`)
7. `_skels/<name>/AGENTS.md` if present (or the matching `CLAUDE.md` for
   Claude Code)

Refresh these files whenever the user notes doc/rule changes or after you make
rule-affecting edits.

## 2. Project Purpose and Junie’s Role

- Makefile-based project generator managing multiple skeletons in
  `_skels/*-skel/`.
- Each skeleton produces a framework-specific service within a wrapper
  project that integrates services around a shared `<wrapper>/.env`
  (DATABASE_URL, JWT_SECRET, ...) and a shared SQLite/Postgres database
  (`<wrapper>/_shared/db.sqlite3` by default).
- **Service directories are named after the service display name** (e.g.
  `"Ticket Service"` → `ticket_service/`). Slug computation lives in
  `dev_skel_lib.slugify_service_name()`. Auto-suffix only when the slug is
  already taken.
- The root `Makefile` orchestrates `gen-*` and `test-*`; `_bin/skel-gen` is a
  relocatable Python entrypoint that delegates to per-skeleton `gen` scripts
  and post-renders the templated `AGENTS.md` shipped with each generated
  service.
- Every `_bin/` CLI is Python and shares helpers via `_bin/dev_skel_lib.py`.
  Update behaviour in that module when you need consistent changes across
  install / update / sync / list / generate flows.
- `_bin/skel-gen-ai` adds an Ollama-backed AI overlay on top of `skel-gen`.
  It runs the interactive dialog FIRST so the chosen service display name
  becomes the on-disk directory before the AI overlay touches anything.
  Manifests live at `_skels/_common/manifests/<skel>.py`. Do not hand-edit
  AI-generated files in test projects — fix the manifest prompt and re-run.
- **Every backend skel** — Python (django, django-bolt, fastapi, flask),
  Java (spring), Rust (actix, axum), and JS (node) — reads
  `DATABASE_URL` / `JWT_SECRET` (and the JVM-friendly
  `DATABASE_JDBC_URL` / `SPRING_DATASOURCE_URL` siblings) from the
  wrapper-level `.env` first then its own local `.env`. Never hardcode
  those values in skel templates. The per-stack helpers are
  `python-dotenv` + helper functions (Python), the `JwtProperties` bean +
  `application.properties` placeholders (Spring), `Config::from_env` +
  `load_dotenv` in `src/config.rs` (Rust), and `dotenv` + `src/config.js`
  (JS).
- Wrapper-level scripts (`./services`, `./run`, `./test`, `./build`,
  `./stop`, `./install-deps`) are generated by
  `_skels/_common/common-wrapper.sh`, source the wrapper `.env`, and
  accept an optional first arg matching a service slug.

Your responsibilities:
1. Maintain generators and skeletons.
2. Keep dependencies supported and reasonably current.
3. Preserve developer workflows and ergonomics described in `_docs/`.

`_test_projects/` holds generated e2e outputs. Never hand-edit it.

## 3. Directory Responsibilities (quick map)

- `Makefile` (root): entrypoint for generator/test targets.
- `_bin/`: Python CLI tools (`install-dev-skel`, `update-dev-skel`,
  `sync-dev-skel`, `skel-list`, `skel-gen`, `skel-gen-ai`) plus
  `dev_skel_lib.py`, `skel_ai_lib.py` (Ollama AI library), and the rsync
  exclude lists.
- `_skels/_common/`: shared assets — `common-wrapper.sh` (wrapper scaffolder),
  `AGENTS.md` / `CLAUDE.md` templates rendered into generated projects, and
  AI manifests under `_skels/_common/manifests/<skel>.py`.
- `_skels/*-skel/`: per-skeleton directories with `Makefile`, `gen`, `merge`,
  `test`, `deps`, `install-deps`, and skeleton-specific docs/rules
  (`AGENTS.md`, `JUNIE-RULES.md`).
- `_docs/`: documentation hub, including `LLM-MAINTENANCE.md` and per-skeleton
  guides.
- `_test_projects/`: generated test apps only. Test Project Location Rule:
  temporary/repro/debug projects must live here, nowhere else.

## 4. Rule Hierarchy and Updates

### 4.1 Apply Rules in This Order

1. Global: this file + `_docs/LLM-MAINTENANCE.md`.
2. Skeleton-specific: `_skels/<name>/JUNIE-RULES.md` (and `_skels/<name>/AGENTS.md`
   if present) plus the skeleton doc in `_docs/`.

Global rules win on conflicts unless the user explicitly overrides them.

### 4.2 Keep Rules in Sync

Update `_docs/JUNIE-RULES.md` and/or `_docs/LLM-MAINTENANCE.md` when global
behaviour, workflows, or maintenance procedures change. Update the relevant
`_skels/<name>/JUNIE-RULES.md` for skeleton-specific changes. Never leave
behaviour changes undocumented.

### 4.3 Mandatory Test Project Location Rule

- All testing/repro/debug projects must be created only under `_test_projects/`.
- Never place such projects at repo root or inside `_skels/`, `_docs/`, or
  service directories. Generate/regenerate them instead of hand-editing.

---

## 5. Maintenance Workflow (default)

Unless the user specifies otherwise:
1. Run generator tests: `make clean-test` then `make test-generators` (all
   skeletons). For a single-skeleton request, run the focused targets noted in
   `_docs/LLM-MAINTENANCE.md` and that skeleton’s rules.
2. Fix every failure with minimal, targeted changes; rerun the same tests until
   green.
3. After tests are green, update rules/docs (global and per-skeleton) to match
   the new behaviour.
4. Keep scope aligned with the requested area (all skeletons vs. one).

---

## 6. Safety and Scope Constraints

1. Do **not** change `SKEL_DIR` detection patterns unless the user asks; see
   `LLM-MAINTENANCE.md` for context.
2. Do **not** commit/modify build artefacts like `node_modules/`, `.venv/`,
   `target/`, or `dist/`.
3. Do **not** edit `_test_projects/` by hand.
4. Prefer small, focused changes; avoid broad refactors without explicit
   approval.
5. Ask the user before any destructive or backward-incompatible action.

---

## 7. Version and Dependency Management

Use `_docs/DEPENDENCIES.md` for policies. Keep skeletons on supported,
reasonably current versions. For major upgrades (FastAPI, Django, Spring, Rust
toolchains, etc.):
1. Review release notes for breaking changes.
2. Adjust skeleton code/scripts accordingly.
3. Run `make clean-test && make test-generators`.

Document global version constraints here or in `_docs/DEPENDENCIES.md`; place
skeleton-specific rules in each skeleton’s `JUNIE-RULES.md`.

---

## 8. Day-to-Day Behaviour

1. Classify tasks as global vs. skeleton-specific.
2. Load all relevant rules files before editing.
3. Follow established patterns in the affected skeleton/tooling.
4. Add or update tests and docs as needed.
5. Keep generator UX intact; avoid weakening tests.

---

## 9. Verification Checklist

- Run relevant `make test-*` targets; for skeleton edits, include skeleton-level
  tests and regeneration of `_test_projects/` as required.
- Confirm docs/rules align with implemented behaviour.
- Ensure `_test_projects/` remains untouched except via regeneration.
